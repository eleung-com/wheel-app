## Project Overview

**Wheel.desk** is a single-page web application for tracking the options wheel strategy (Cash-Secured Puts → assignment → Covered Calls). It is a zero-dependency, no-build, single-file HTML app.

The two key files are:
- `wheel-app/index.html` — the current production version
- `wheel-app/wheel` — an alternate/earlier version (same structure)

## Running the App

Open `wheel-app/index.html` directly in a browser — no build step, no server required. For development, a simple local HTTP server avoids CORS issues with Yahoo Finance fetches:

```bash
cd wheel-app && python3 -m http.server 8080
# then open http://localhost:8080
```

## Architecture

Everything lives in one HTML file: CSS in `<style>`, markup in `<body>`, and all JavaScript in a single `<script>` block at the bottom. There are no modules, no bundler, no framework.

### Data layer — Google Sheets as backend

User data (watchlist, positions, criteria, signals) is persisted in a Google Apps Script web app acting as a simple JSON store. Credentials (sheet URL + secret) are stored in `localStorage` — never in source. The three sync functions are:

- `sheetRead()` — GET with `?action=read`
- `sheetWrite()` — POST with JSON body (some CORS environments block this)
- `sheetWriteViaGet()` — GET with URL-encoded payload (primary write path, avoids CORS)

All mutations (add/edit/delete position, add/remove watchlist ticker) call `sheetWriteViaGet()` immediately after updating in-memory state.

### In-memory state

All runtime data lives in the global `S` object:

```js
S = {
  watchlist: [],   // { ticker, addedAt, liveData }
  positions: [],   // { id, ticker, type, qty, cost, strike, expiry, prem, curPrem, notes, enteredAt }
  signals:   [],   // generated by runScreener()
  criteria:  {},   // screening thresholds (IVR, RSI, Stoch, MA, delta, DTE, etc.)
  lastRefresh, editId
}
```

### Market data — Yahoo Finance

`fetchQ(ticker)` hits `query1.finance.yahoo.com/v8/finance/chart` for 1y daily OHLCV data and computes all indicators client-side:
- **IVR estimate**: HV30 × 1.25 + distance-from-52w-high × 0.15
- **RSI-14**: simple 14-period, last 16 closes
- **Stochastic %K**: 14-period high/low range
- **MA check**: configurable period (default 200), falls back to available history

`fetchOptionPrice(ticker, type, strike, expiry)` hits `query1.finance.yahoo.com/v7/finance/options` with ±1 day date offsets and fallback to scanning all expiry dates. Uses bid/ask midpoint, falls back to lastPrice.

### Signal engine

`runScreener()` fetches all unique tickers (watchlist + positions), then generates four signal types:
- **CSP** (`csp`): watchlist ticker passes IVR ≥ threshold, RSI ≤ threshold, Stoch ≤ threshold, price above MA
- **Covered Call** (`cc`): position has ≥100 shares and IVR ≥ `ccIvr`, no existing short call
- **Roll** (`roll`): short option strike is breached (put: price < strike; call: price > strike)
- **Close Early** (`close`): premium captured ≥ `closePct`% AND time elapsed ≥ `closeDtePct`%

Partial CSP signals (2–3/4 criteria met) are shown in a separate section. Auto-refresh runs every 20 min during NY market hours.

### UI patterns

- **Tabs** (top nav) + **bottom nav** for mobile — controlled by `switchTab()` / `switchPage()`
- **Modals** use an `.overlay` wrapper; clicking the overlay backdrop closes them
- **CSS custom properties** define the full dark theme palette (green `--g`, amber `--a`, red `--r`, blue `--bl`, purple `--pu`)
- **IBM Plex Mono** for all numeric/ticker display; **DM Sans** for UI text
- Responsive breakpoint at 640px switches to 3-column grids and centered layout

### Position rendering

Shares are grouped by ticker with weighted-average cost basis. Options are rendered in a separate table with live P&L computed as `(prem - effectiveCurPrem) × qty × 100`. `effectiveCurPrem` prefers `_liveCurPrem` (fetched during screener run, stored transiently on the position object) over manually-entered `curPrem`.
